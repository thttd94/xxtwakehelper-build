- User paused this topic and explicitly asked to save context to memory.
- Environment: iKuai main router gives WAN/LAN-side IP `192.168.8.4` to Genrouter `192.14.0.1` over `eth0` DHCP. Genrouter WAN status from OpenWrt: `eth0 inet 192.168.8.4/22`, default route `via 192.168.9.1`, DHCP server/gateway `192.168.9.1`, DNS `8.8.8.8/8.8.4.4`. Genrouter LAN is `192.14.0.1/20`, clients are `192.14.4.x`, `192.14.5.x`, etc.
- User’s actual requirement: clients are mainly for TikTok downloads/events; each client must use SOCKS5/HTTP proxy and TikTok must not see the real WAN IP. Need WebRTC/leak prevention so TikTok sees a Japan/proxy IP as if real. Speed should be realistic relative to WAN, but proxy/core limitations matter.
- iKuai screenshots showed `192.168.8.4` (Genrouter) with high traffic, e.g. Rx around `37.28 MB/s` (~298 Mbps) and connection count around `1920`. iKuai overall speed test also showed high WAN capacity (hundreds of Mbps). Conclusion: iKuai/uplink is not the immediate bottleneck.
- iKuai terminal details for `192.168.8.4` showed many connections labeled `SOCKS`, many TCP sessions, destinations including port `1080`, and interfaces like `wan2` / `adsl_TT01`. Conclusion: Genrouter traffic is largely going through proxy/SOCKS path, not plain direct NAT.
- On Genrouter `192.14.0.1`, mangle rules showed heavy TPROXY design: `-A PREROUTING -i br-lan -j GENROUTER`, then hundreds of per-client rules like `-A GENROUTER -s 192.14.4.x/32 -p tcp -j TPROXY --on-port 9888 --tproxy-mark 0x4d2/0x4d2`. This means many client TCP flows are transparently intercepted into gencore/proxy port `9888`.
- A temporary bypass test was performed safely: saved iptables/ip rules under `/tmp/genrouter_tproxy_bypass_test/`, removed only the PREROUTING hook into GENROUTER, and installed `/tmp/genrouter_tproxy_restore.sh` with auto-restore after 120 seconds. Router remained reachable afterward; later probe showed `192.14.0.1` ports `886`, `22`, `9000`, `9001`, `8000` open. Observed iKuai Rx rate increased in screenshots during/around bypass (~90 Mbps earlier to ~150 Mbps, later ~298 Mbps), suggesting TPROXY/gencore path affects throughput, but not enough data for a final controlled benchmark.
- Concept explained to user: TPROXY is not a standalone app but Linux kernel/iptables transparent proxy mechanism plus policy routing that diverts client traffic into a userspace proxy core (here gencore/port 9888), then to SOCKS/HTTP proxy outbound. NAT direct is faster but would expose real WAN IP to TikTok, so not acceptable for TikTok-safe clients.
- Recommended future direction: do not simply disable TPROXY. Build/implement a “TikTok Safe Proxy Mode”: for TikTok clients, TCP goes through proxy; direct UDP/WebRTC/STUN/QUIC/DNS leak paths are blocked; no direct WAN for those clients; optimize TPROXY rules using ipset/nft/subnet instead of hundreds of per-IP rules; split clients across multiple strong Japan proxies; test/health-check proxy upstreams.
- Leak prevention targets discussed: block or control direct DNS (UDP/TCP 53), UDP 443/QUIC, STUN/TURN ports like UDP 3478/5349, and possibly direct UDP for TikTok-safe clients unless a safe UDP proxy path exists.
- User was confused by options; simplified explanation given: if clients go direct/NAT, speed is higher but TikTok sees real WAN IP. To make TikTok see Japan IP, traffic must go through Japan proxy and all leak side doors must be blocked. Speed then depends on proxy strength, gencore/TPROXY overhead, rule efficiency, and load distribution across proxies.

## Genrouter versioning / Git upload / local backup conventions
- User explicitly asked to remember how Genrouter versions should be named, uploaded to Git, and stored locally inside Genrouter.
- Version keys must be normalized without a leading `V`: use `1.1`, `2.2`, `10.01`, `200`, etc. Commands should accept `V2.2` but scripts must strip leading `V/v` and store/search as `2.2`. User’s expected command style remains `sh /opt/proxy-manager-v1/rollback_version.sh 2.2` (or `1.1`, `10.01`, etc.).
- Source of truth for online updates is GitHub repo `thttd94/GEN`, branch `main`. When user says “đẩy lên Git” / “đẩy bản vá lên GIT”, changes must be committed and pushed to this repo because routers update by downloading the GitHub main tarball.
- Online Git main tarball should be a lightweight bootstrap/package, not a huge embedded full-system snapshot, because small router rootfs can run out of space when downloading/extracting under `/root`. Use a large workdir when updating routers: `cd /mnt/nvme0n1p4 2>/dev/null || cd /data 2>/dev/null || cd /tmp` before downloading Git.
- Local version backup stores inside Genrouter:
  - `/data/genrouter_backups/versions/<ver>/package`
  - `/data/genrouter_backups/versions/<ver>/system`
  - fallback if no `/data`: `/root/genrouter_backups/versions/<ver>/package` and `/root/genrouter_backups/versions/<ver>/system`
- Never delete old version backups. Backup stores themselves must be excluded from snapshots to prevent recursive/infinite backups: `/root/genrouter_backups` and `/data/genrouter_backups`.
- Full-version backup requirement: a real version restore must restore full Genrouter/router state, not just `/opt/proxy-manager-v1`. Snapshot should include at least `/opt/proxy-manager-v1`, `/etc/genrouter`, `/etc/config`, `/etc/init.d`, `/etc/rc.local`, `/etc/crontabs`, `/etc/shm`, plus a `system_manifest.txt` with mount/df/blkid/lsblk/fdisk/ip/services metadata.
- Full snapshot script path in repo/package: `full_system_backup.sh`. Example to create local full snapshot for version `200`: `sh /opt/proxy-manager-v1/full_system_backup.sh 200`. It should create both `package/` and `system/` under the selected backup root.
- Rollback script path: `rollback_version.sh`. Restore order should be system tarballs first, then clean-copy `package/` into `/opt/proxy-manager-v1` as final source of truth, then restart/enable relevant services.
- Online full restore architecture: Git `main` contains bootstrap scripts such as `restore_version_online.sh` and `rollback_version.sh`; large full artifacts should live separately (e.g. artifacts branch/release URL), then `restore_version_online.sh` downloads artifact to a large partition, seeds `/data` or `/root` backup store, and invokes rollback.
- V2.2 artifact used during this session: `https://raw.githubusercontent.com/thttd94/GEN/artifacts/genrouter-v2.2-full.tar.gz`. Git commits around this work: `fe34a73 Add online full artifact restore script`, `86663ca Preserve target network during full restore`, `f682d35 Verify runtime after full restore`, `655b736 Keep access services alive after restore`.
- Full restore safety: default `PRESERVE_NETWORK=1` to preserve target router’s `/etc/config/network`, `/etc/config/dropbear`, `/etc/config/firewall`, and `/etc/dropbear` so restoring from a source router does not change the target LAN IP or kill SSH access. Do not reload/restart network by default; only if explicitly requested via something like `RELOAD_NETWORK_AFTER_RESTORE=1`.
- Runtime verification after restore must be mandatory. Do not print OK just because files were restored. Verify SSH/dropbear port `886`, Genrouter core/API `9000`, proxy-manager web/API `9001`, and relevant API calls such as `http://127.0.0.1:9000/api/router/info` and `http://127.0.0.1:9001/api/pm/router-info`. If verification fails, return error instead of claiming success.
- Current recommended online restore command pattern for V2.2 from a router:
  `cd /mnt/nvme0n1p4 2>/dev/null || cd /data 2>/dev/null || cd /tmp; rm -rf GEN GEN-main GEN-main.tar.gz; wget --no-check-certificate -O GEN-main.tar.gz "https://codeload.github.com/thttd94/GEN/tar.gz/refs/heads/main"; tar -xzf GEN-main.tar.gz; mv GEN-main GEN; cd GEN; chmod +x restore_version_online.sh rollback_version.sh install.sh full_system_backup.sh setup_data_disk.sh; GEN_FULL_URL="https://raw.githubusercontent.com/thttd94/GEN/artifacts/genrouter-v2.2-full.tar.gz" PRESERVE_NETWORK=1 SKIP_PRE_ROLLBACK=1 sh restore_version_online.sh 2.2`
- For normal lightweight Git update/install without full artifact, use same large workdir pattern then run `sh install.sh`; but do not call that a full restore unless full `system/` backup/artifact is present.
